Fencing Results Reader PHP / Data parsing / Analytics

A web tool I built for the French Fencing Federation to automatically read competition result files (XML exports), extract key statistics, and present them in a clear, coach-friendly dashboard. This replaces hours of Excel work after each event.

01. Context Before the tool

After each fencing competition, the federation receives raw result files with all bouts, pools, eliminations, rankings, and referees. Before this project:

  • Staff had to open these files manually and search for relevant info (who beat who, score spreads, cards, etc.).
  • They copied data into Excel to rebuild standings and produce internal reports for coaches.
  • Every competition meant starting over. The process depended on whoever was on duty that weekend.

They asked me to make this fast, reliable, and usable by non-technical staff.

02. Objective What I delivered

I developed a PHP web interface that:

  • Parses official competition exports (like lille.xml).
  • Displays pools (poules.php), final rankings (classement_f.php), match sheets (match.php), registered athletes (inscrits.php), and referees (arbitre.php).
  • Computes performance indicators (victories, score differences, etc.).
  • Shows results in a clean HTML view that can be shared instantly with coaches and managers.

I demoed it live to the federation‚Äôs manager, collected feedback on what stats mattered most, and iterated on wording and layout.

03. Technical Approach How it works

  • Central data source: A single XML file (example: lille.xml) contains the full competition: pools, fencers, referees, match results, ranking.
  • PHP pages: each PHP file focuses on one view: poules.php shows pool tables, match.php shows match details, classement_f.php shows standings, inscrits.php lists competitors, arbitre.php lists referees.
  • Rendering: data is turned into styled HTML tables using a shared stylesheet (styles.css) so it‚Äôs readable for non-tech users.
  • No DB: everything is computed in memory from the XML ‚Üí easy to deploy, easy to move to another competition.

This structure means you can just drop a new competition XML file in the folder, and the site instantly works for that event ‚Äî no recoding.

04. Tech Stack Tools

  • PHP for logic, parsing, stats calculation
  • SimpleXML for reading the exported federation XML files
  • HTML / CSS for rendering clean tables for coaches and referees
  • Modular pages (poules.php, match.php, etc.) so each part of the competition has its own URL

05. Impact Why it matters

After this tool:

  • They can generate a competition summary in minutes instead of manually rebuilding Excel sheets.
  • All coaches see the same standardized view (no more ‚Äúmy version vs your version‚Äù).
  • Discussions after competition focus on performance, not on data cleaning.

It also trained me to work with non-technical stakeholders (federation staff, referees), translate ‚Äúwe need stats‚Äù into computable KPIs, and design around how they actually talk about fencing.

06. Needs Analysis Functional specification

Before touching code, I produced a formal Functional Specification (‚Äú√âtat des Besoins‚Äù) that described all screens and all data the federation expected to see. This is basically me acting like a product owner.

  • Home / index.php: competition name, organizer, location, date, fencing formula (e.g. Pools ‚Üí Direct Elimination), and live status.
  • inscrits.php: list of registered fencers with name, club, category, weapon. Searchable.
  • arbitre.php: list of referees assigned to the event (name, club, qualification level).
  • poules.php: pool tables with every match, hits scored / taken, victories, referee on that pool.
  • classement_f.php: intermediate and final ranking with clear medal colors and elimination round reached.
  • match.php: detailed view of a specific match or tableau branch for review with the coach.

The spec also lists planned improvements:

  • Track cards / penalties per fencer.
  • Referee stats to support evaluation of officials.
  • Highlight ‚Äúunexpected upsets‚Äù (lower seed beats top seed).

07. Core PHP Logic XML parsing

Below is a simplified version of the logic I use (in poules.php) to read the official XML export (example: lille.xml), loop through each pool, and list the bouts with scores. 


// Load competition data
$xml = simplexml_load_file("lille.xml");

// Store all pools
$pools = [];

foreach ($xml->Poule as $poule) {
    $poolNum = (string)$poule['Num'];

    // Keep structured info for this pool
    $pools[$poolNum] = [
        "referee" => (string)$poule['Arbitre'],
        "matches" => []
    ];

    // Loop over each match in the pool
    foreach ($poule->Match as $match) {
        $fencerA = (string)$match->TireurA['Nom'];
        $fencerB = (string)$match->TireurB['Nom'];
        $scoreA  = (int)$match['ScoreA'];
        $scoreB  = (int)$match['ScoreB'];

        $pools[$poolNum]["matches"][] = [
            "a" => $fencerA,
            "b" => $fencerB,
            "sa" => $scoreA,
            "sb" => $scoreB
        ];
    }
}

// Example rendering (HTML table for each pool)
foreach ($pools as $num => $data) {
    echo "<h2>Pool " . $num . " (Ref: " . $data["referee"] . ")</h2>";
    echo "<table>";
    echo "<tr><th>Fencer A</th><th>Fencer B</th><th>Score</th></tr>";

    foreach ($data["matches"] as $m) {
        $scoreLine = $m["sa"] . " - " . $m["sb"];
        echo "<tr>";
        echo "<td>" . $m["a"] . "</td>";
        echo "<td>" . $m["b"] . "</td>";
        echo "<td>" . $scoreLine . "</td>";
        echo "</tr>";
    }

    echo "</table>";
}

This pattern is reused across pages: classement_f.php loops through final standings, arbitre.php loops through referees, inscrits.php loops through fencer registrations, and so on. The federation can simply change the XML file, and the interface instantly reflects a new competition.
